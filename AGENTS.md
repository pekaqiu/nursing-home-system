# AGENTS.md

## 一、项目定位

本项目是本科毕业设计级别的《融合适老化交互设计的养老院信息管理系统设计与实现》。

本项目不是企业级养老院综合业务平台，不追求功能多，不追求复杂架构，不追求商业级高并发或高可用。

项目目标是：在 JSP、JDBC、MySQL、Tomcat 技术栈下，实现一个结构清晰、功能闭环完整、页面全中文、具备适老化交互特点、能够用于论文撰写和毕业答辩演示的养老院信息管理系统。

所有开发工作必须服务于以下目标：

1. 项目能够正常运行。
2. 数据能够正常写入和读取 MySQL。
3. 三类角色边界清楚。
4. 核心业务闭环完整。
5. 页面适合论文截图和答辩演示。
6. 系统复杂度控制在本科毕业设计可完成范围内。

## 四、系统复杂度控制

本项目功能必须精简。

不要为了“看起来高级”而增加不必要模块。

不要把系统做成完整养老院商业平台。

不要引入高并发、高可用、云端部署、复杂数据大屏、复杂统计图表、复杂财务系统等内容。

本科毕业设计最重要的是：

1. 功能闭环清楚。
2. 数据库设计合理。
3. 页面可以演示。
4. 代码结构能讲清楚。
5. 论文中能画出功能结构图、E-R 图、数据流程图和程序流程图。

## 五、保留的系统角色

本系统只保留三类角色：

1. 普通用户
2. 工作人员
3. 管理员

角色区分只需要使用用户表中的 `role` 字段。

建议角色值：

1. `USER`：普通用户
2. `STAFF`：工作人员
3. `ADMIN`：管理员

禁止设计复杂权限系统。

不需要设计权限表、菜单权限表、角色权限关联表。

## 六、保留的核心功能

系统只保留以下核心功能。

### 1. 普通用户端

普通用户端只保留 4 个功能：

1. 注册登录
2. 查看养老院信息
3. 查看公告资讯
4. 提交入住申请

普通用户端不做以下功能：

1. 床位与收费信息查看
2. 在线咨询
3. 个人中心
4. 健康档案查看
5. 在线缴费
6. 消息通知
7. 文件上传

### 2. 工作人员端

工作人员端只保留 2 个功能：

1. 老人入住登记
2. 日常护理记录管理

工作人员端不做以下功能：

1. 独立健康监测信息录入
2. 活动安排管理
3. 来访登记管理
4. 异常情况上报处理
5. 在线沟通管理
6. 排班管理

说明：健康情况可以合并到护理记录中的“身体情况”或“护理备注”字段，不单独设计健康监测模块。

### 3. 管理员端

管理员端只保留 4 个功能：

1. 用户信息管理
2. 老人档案管理
3. 床位信息管理
4. 入住申请审核

管理员端不做以下功能：

1. 护理信息管理
2. 收费管理
3. 公告资讯管理
4. 轮播图管理
5. 复杂系统权限管理
6. 数据大屏
7. 财务报表

说明：公告资讯可以作为普通用户端展示内容存在，但不要求后台做复杂公告管理。若项目已有简单公告表，可以保留基础展示；如果没有，不要为了公告管理单独扩展复杂后台。

## 七、核心业务闭环

本系统最终只需要形成以下业务闭环：

普通用户注册登录  
→ 普通用户查看养老院信息和公告资讯  
→ 普通用户提交入住申请  
→ 管理员审核入住申请  
→ 工作人员进行老人入住登记  
→ 管理员或工作人员维护老人档案和床位状态  
→ 工作人员记录日常护理情况

这个闭环已经足够支撑本科毕业设计，不要继续扩展其他业务链条。

## 八、建议保留的数据库表

数据库只建议保留以下核心表：

1. `sys_user`：用户表
2. `elderly_profile`：老人档案表
3. `checkin_apply`：入住申请表
4. `bed_info`：床位信息表
5. `care_record`：护理记录表
6. `notice`：公告表

用途：保存普通用户、工作人员、管理员账号。

建议字段：

1. `id`：主键
2. `username`：用户名
3. `password`：密码
4. `real_name`：真实姓名
5. `phone`：手机号
6. `role`：角色
7. `status`：状态
8. `create_time`：创建时间
9. `update_time`：更新时间

### 2. elderly_profile 老人档案表

用途：保存入住老人基本档案。

建议字段：

1. `id`：主键
2. `name`：老人姓名
3. `gender`：性别
4. `age`：年龄
5. `phone`：联系电话
6. `family_contact`：家属联系人
7. `family_phone`：家属电话
8. `health_condition`：健康状况
9. `checkin_date`：入住日期
10. `bed_id`：床位 id
11. `status`：状态
12. `create_time`：创建时间
13. `update_time`：更新时间

### 3. checkin_apply 入住申请表

用途：普通用户提交入住申请，管理员审核。

建议字段：

1. `id`：主键
2. `user_id`：申请用户 id
3. `elderly_name`：老人姓名
4. `elderly_age`：老人年龄
5. `gender`：性别
6. `contact_phone`：联系电话
7. `health_desc`：健康说明
8. `apply_reason`：申请说明
9. `audit_status`：审核状态
10. `audit_remark`：审核备注
11. `create_time`：创建时间
12. `update_time`：更新时间

### 4. bed_info 床位信息表

用途：管理养老院床位。

建议字段：

1. `id`：主键
2. `room_no`：房间号
3. `bed_no`：床位号
4. `bed_type`：床位类型
5. `status`：状态
6. `remark`：备注
7. `create_time`：创建时间
8. `update_time`：更新时间

说明：精简版可以不做收费字段。如果项目已有 `price` 字段，可以保留但不作为核心功能展示。

### 5. care_record 护理记录表

用途：工作人员记录日常护理情况。

建议字段：

1. `id`：主键
2. `elderly_id`：老人 id
3. `staff_id`：工作人员 id
4. `care_content`：护理内容
5. `body_condition`：身体情况
6. `care_time`：护理时间
7. `remark`：备注
8. `create_time`：创建时间
9. `update_time`：更新时间

### 6. notice 公告表

用途：普通用户端展示公告资讯。

建议字段：

1. `id`：主键
2. `title`：标题
3. `content`：内容
4. `publish_time`：发布时间
5. `status`：状态
6. `create_time`：创建时间
7. `update_time`：更新时间

## 十一、适老化界面要求

普通用户端必须体现适老化设计。

适老化设计至少包括：

1. 字体偏大。
2. 按钮明显。
3. 页面颜色对比清晰。
4. 表单输入框清楚。
5. 操作步骤少。
6. 提示语使用中文。
7. 错误提示明确。
8. 常用入口放在显眼位置。
9. 页面不要堆叠过多信息。
10. 不使用复杂动画。

工作人员端和管理员端以功能清晰、表格可维护、表单易填写为主，不需要做复杂视觉效果。

## 十二、代码结构建议

如果项目采用传统 JSP + Servlet + JDBC 结构，建议保持以下分层：

1. `entity` 或 `model`：实体类
2. `dao`：数据库访问层
3. `service`：业务逻辑层
4. `servlet` 或 `controller`：请求处理层
5. `jsp`：页面展示层
6. `util`：数据库连接等工具类

不要因为追求“规范”而大规模重构已有可运行代码。

如果已有项目结构与上述略有不同，但能运行、能讲清楚，可以保留。

## 十三、开发顺序

必须按照以下顺序推进：

1. 先保证项目能启动。
2. 再保证数据库能连接。
3. 再保证登录注册能使用。
4. 再保证三类角色能跳转到不同首页。
5. 再实现入住申请和审核闭环。
6. 再实现老人档案和床位管理。
7. 再实现工作人员入住登记和护理记录。
8. 最后统一页面中文文案和适老化样式。

不要一开始就美化页面。

不要一开始就扩展新模块。

不要一开始就重构全项目。
